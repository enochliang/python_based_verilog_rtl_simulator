# RTL Fault Simulator

The **RTL Fault Simulator** is a specialized framework designed for fault simulation and reliability analysis of Register Transfer Level (RTL) circuit designs. By performing static analysis on XML Abstract Syntax Trees (AST) generated by Verilator and combining it with dynamic logic value simulation, the tool calculates fault propagation across registers under real-world workloads. The final output includes **ACE (Architecturally Correct Execution)** analysis reports.

| Feature | Description |

| **Target Language** | Verilog / SystemVerilog RTL  |
| **AST Source** | Verilator XML output (`--xml-output`)  |
| **Simulation Granularity** | single cycle register fault propagation |
| **Analysis Method** | Static AST parsing + Dynamic Fault List Propagation |
| **Hardware Targets** | picorv32, tinyriscv, AES, SHA1, FFT, SPI, etc. |
| **Main Outputs** | RW Table, ACE Analysis results, Reduced Control Dependency Graph |

---

## System Architecture & Data Flow

The framework operates through a **three-phase pipeline** to transform raw Verilog code into actionable reliability data:


### Phase 1: AST Preprocessing
* **Input:** Verilator XML AST.
* **Operations:** Removes non-essential nodes (integer, initial, parameter), merges aliased signals, marks signal types (register/wire), and performs **topological sorting** of sub-circuits.
* **Output:** A "clean" scheduled XML AST and sub-circuit order list.

### Phase 2: Python AST Construction
* **Input:** Scheduled AST from Phase 1.
* **Operations:** Converts XML nodes into custom Python objects (`Verilog_AST_*`), flattens array signals, and loads cycle-by-cycle logic values.
* **Output:** A `Verilog_AST` object tree ready for simulation.

### Phase 3: Simulation & Analysis
* **Input:** Object tree, logic values, and RW Table.
* **Operations:** Traverses sub-circuits based on the schedule, calculates node outputs using a 3-state logic library, propagates Fault Lists, and generates ACE/un-ACE statistics.
* **Output:** ACE results (JSON), RW Table (CSV), and control reduction reports.

## Core Technical Concepts

### Three-State Signal Model
Each `Verilog_AST_Circuit_Node` maintains three distinct value fields to track state across clock cycles:
* **`value`**: Current cycle combinational logic value.
* **`cur_value`**: Latched register value at the start of the cycle.
* **`next_value`**: Updated register value at the end of the cycle, which becomes the next cycle's `cur_value`.

### Propagation Probability Calculation
The tool calculates propagation probability using the following logic:
* **Mask Probability:** $$mask\_probs = [1 - prob \text{ for } (dst\_reg, prob) \in (write\_events \cup ctrl\_events)]$$ 
* **Total Mask Probability:** $$tot\_m\_probs = \prod mask\_probs$$ 
* **Final Propagation Probability:** $$tot\_p\_probs = 1 - tot\_m\_probs$$

## Quick Start Guide

### 1. Requirements
* [cite_start]Python 3.9+
* [cite_start]`lxml`, `pandas`
* [cite_start]Verilator 4.x/5.x

### 2. Generate AST
[cite_start]Use Verilator to produce the XML representation of your design[cite: 120]:
```bash
verilator -xml-output --bbox-sys -Wall --top-module <TopModule> <Design>.v
```

### 3. Run Preprocessing
```bash
python ast_schedule.py -f obj_dir/V<TopModule>.xml
```

### 4. Execute Fault Simulation
```python
from ast_sim_prepare import SimulatorPrepare
# Initialize and load values
sim = SimulatorPrepare(ast, logic_value_file_dir='./logic_values/')
sim.preprocess()
sim.load_logic_value(cycle=5)
sim.init_fault_list()
```

## Known Issues & Limitations
* **`rtl_functions.py`**: `val_neq()` contains dead code [cite: 160][cite_start], and `val_sel()` has a `NameError` (undefined `width` variable) when `start_bit == 0`.
* **`ast_node_define.py`**: The `_children` attribute is currently a class variable; it should be initialized in `__init__` to avoid shared state between instances.
* **`rw_split.py`**: Currently an empty skeleton module.
* **Naming**: `dataloader.py` uses regex to strip `genblk` prefixes, which may require updates for complex nested generate blocks.
